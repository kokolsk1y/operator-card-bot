# Мониторинг цен на маркетплейсах

Ищет на маркетплейсах наши товары дешевле нашей закупочной и пишет в Telegram.
Отдельный бот и отдельный контейнер — генератор карточек (`../index.mjs`) не трогает.

## Как работает

Два ритма, и это осознанно разные вещи:

| | Как часто | Почему так |
|---|---|---|
| **Поиск кандидатов** | раз в сутки | Дорого и мусорно. WB отдаёт `429` уже на 4-м запросе подряд. |
| **Проверка цен** | раз в 20 мин | Дёшево и точно: 60 товаров за один запрос по подтверждённым ID. |

Между ними — **человек**. Бот находит кандидатов, присылает карточку с кнопкой
«Это оно?», и только подтверждённое попадает в отслеживание.

### Почему без человека нельзя

Поиск маркетплейса **не умеет** в характеристики. Проверено на живой выдаче
2026-07-16, запрос `Лампа LED A60 E27 3000K 11Вт 990Lm IEK`:

- вернулось **100 товаров**, ни одного «не найдено» — WB всегда набивает страницу;
- первые 33 позиции — лампы другой мощности, цоколя, формы колбы;
- единственное верное совпадение стояло **34-м**.

То есть «брать топ-10 самых совпадающих» единственную реальную находку бы **не увидело**.
Поэтому отбирает `match.mjs`, а не релевантность площадки: из 100 товаров он отсеял 94.

### Почему «раз в день» мало

Цена той же лампы (`nmId 197050934`) за час менялась **63 → 80 ₽ (+27%)**.
Ошибочные цены живут минуты. Раз в сутки их гарантированно проспишь.

## Запуск

```bash
npm install
npm test          # 33 теста, сеть не нужна
npm run smoke     # живая проверка WB: поиск → отсев → цены
npm start
```

Настройки — переменные окружения, см. `../.env.example`.
Без `PRICES_ALLOWED_USERS` сервис не стартует: он знает закупочные цены компании.

## Грабли, на которые уже наступили

**Кириллические омоглифы.** Продавцы пишут `Е27` кириллицей и `E27` латиницей
вперемешку, часто в одном названии. Визуально не отличить, для сравнения — разные
строки. То же с `А60`/`A60`, `3000К`/`3000K`. Лечится в `normalize.mjs`: верхний
регистр, затем свёртка 12 заглавных омоглифов в латиницу.

**Связки.** «2шт» за 390 ₽ — это 195 ₽/шт. Без деления на количество бот
сравнивает цену лота с ценой штуки и врёт кратно. `extractPack()`.

**НДС.** Делить цену на 1.22 «потому что НДС 22%» — прямой убыток: у продавца
на УСН НДС в цене нет вообще. Вычитаем **только** при явном признаке возврата
с витрины, ставку берём оттуда же (бывает 0/5/7/10/22), а не константой.

**`fetch()` не работает с `card.wb.ru`.** undici шлёт свои заголовки и имеет свой
TLS-отпечаток → `403`, тогда как curl получает `200`. Лечится `node:https` +
`ciphers: 'DEFAULT'`. Забавно, что «притвориться Chrome» тут **вредит**: curl_cffi
с `impersonate=chrome` тоже получает `403`.

**Оборванная цепочка middleware.** В `../index.mjs` обработчик текста —
catch-all без `next()`, поэтому всё зарегистрированное после него молча мертво.
Здесь каждый «не про меня» обработчик обязан вызвать `next()`; тест на это есть.

## Ozon — почему выключен

`adapter.enabled()` = `false`, пока `PRICES_OZON_ENABLED=1` не выставлен вручную.

Установлено (с IP немецкого датацентра): Ozon отвечает `403` с заголовком
`ozon-antibot: 1` и текстом «Выключите VPN, перезагрузите роутер или подключитесь
к другой сети». Режет **репутация IP**, а не признаки робота — маскировка
не помогает и не нужна.

Не установлено:

1. пустит ли антибот автоматический клиент с **российского** IP;
2. виден ли бейдж **«Возврат НДС N%»** без входа в аккаунт — он живёт на витрине
   `ozon.ru/business`, и именно он сообщает ставку и право на вычет;
3. структура ответа `composer-api` — он отдаёт виджеты с JSON внутри JSON,
   по памяти такое не разбирают.

Снять всё разом:

```bash
node prices/probe-ozon.mjs    # с российского IP, лучше на сервере
```

Если бейдж требует логина — это отдельное решение, а не деталь: бот пойдёт под
**закупочным аккаунтом компании**, и под баном окажется он, а не тестовый IP.

## Что бот не проверяет

**Маркировку «Честный знак».** С 1 мая 2026 электротовары (розетки, выключатели,
кабели, LED-лампы) маркируются обязательно. Купленный в рознице маркированный
товар выводится из оборота — перепродать его легально нельзя, нужны коды
маркировки в УПД через ЭДО. На карточке это не видно, поэтому бот и не пытается:
проверка на человеке. Решение принято осознанно 2026-07-17.
